## 1. Project Overview

Yaad (याद) is a **model-agnostic, graph-native memory system** for coding agents. Written in Go (go 1.25.5), it's designed as a single binary with zero runtime dependencies.

### Key Stats
| Metric | Value |
|--------|-------|
| **Go source files** | 47 |
| **Packages** | 28 |
| **Test functions** | 38 (integration_test.go) |
| **Test pass rate** | 100% (38/38) |
| **Lines of Go code** | ~9,500 (excluding vendor) |
| **Binary size** | ~18MB |
| **External Go deps** | 12 major (cobra, mcp-go, grpc, sqlite, bubbletea, toml, uuid) |
| **SDK languages** | Go, Python, TypeScript (partial) |
| **CLI commands** | 27+ cobra commands |

### Architecture Layers
```
cmd/yaad/         → CLI entrypoint (cobra commands)
internal/storage/ → SQLite persistence layer (FTS5, embeddings, sessions, versions)
internal/graph/   → DAG operations (cycle detection, BFS, intent-aware traversal, impact)
internal/engine/  → Business logic (remember, recall, context, entities, decay, proactive, feedback)
internal/server/  → 4 protocol servers (MCP, REST, gRPC, SSE)
internal/*/       → 15+ supporting packages (privacy, hooks, git, sync, team, skill, etc.)
sdk/              → Go SDK (embeddable), Python SDK (REST client), TypeScript SDK (REST client)
```

## 5. Quality Issues

### Bugs

1. **`main.go` line 267-300**: `embedCmd` references `embeddings.NewLocal()` which returns a stub provider. The `hybrid-recall` and `proactive` commands also use this stub — vector search will return random pseudo-vectors.

2. **`graph/graph.go` lines 82-93 — Cycle detection flaw**: The `checkCycle` function checks if adding fromID→toID creates a cycle among acyclic edges by walking ancestors of `fromID` and checking if `toID` is among them. However, it only checks one direction. A cycle `A→B`, `B→A` would be caught, but what about `A→B→C`, then adding `C→A`? The recursive CTE walks `fromID` (C)→ancestors→A, B... Yes, this should work, but the query is:
   ```sql
   SELECT e.from_id FROM ancestors a JOIN edges e ON e.to_id = a.id AND e.acyclic = 1
   ```
   This only follows the `to_id` direction (edges pointing TO ancestors), which is correct for ancestors. But if the new edge is C→A, then fromID=C, toID=A. It checks if A is an ancestor of C (via acyclic edges). This looks correct.

3. **`engine/memory.go`** - The `sortByScore` function uses a bubble sort (O(n²)). For small n this is fine, but with 1000+ nodes it's wasteful.

4. **`engine/search.go` — `vectorSearch` reads ALL embeddings into memory**: `store.AllEmbeddings()` loads every vector into a map, then computes cosine similarity against every one. This is O(n) memory and CPU, not scalable.

5. **`server/grpc.go` — JSON gRPC workaround**: The gRPC implementation uses a custom ServiceDesc with JSON-encoded messages. This means gRPC clients expecting protobuf-serialized data won't work. The proto file exists (`internal/proto/yaad.proto`) but no protobuf-generated Go code was found.

### Incomplete Error Handling

1. **`graph/graph.go` line 115-117**: In `ExtractSubgraph`, when `GetNode` fails (missing node), the error is silently ignored: `continue // skip missing nodes`. This could mask real bugs.

2. **`engine/memory.go` — `logErr` pattern**: Many operations use `logErr("op", err)` which just logs and continues. The temporal link, conflict resolution, and entity linking all use this fire-and-forget pattern. While intentional (graceful degradation), it can mask failures.

3. **`hooks/runner.go`**: `PostToolUse` silently reads session ID from a file — if the file doesn't exist, sessionID is empty-string, meaning nodes are created without session association.

4. **`ingest/dual_stream.go`**: Slow path silently drops when queue is full (256 cap). Graceful degradation, but no warning or metrics.

### Code Quality Issues

1. **Single test file**: All 38 tests in `integration_test.go` — this is 2,300+ lines. Should be split into package-level `_test.go` files.
2. **No interfaces for storage/graph**: Makes mocking impossible. The engine is tightly coupled to concrete storage and graph types.
3. **Bubble sort in production code**: `sortByScore` in `engine/memory.go` uses bubble sort instead of `sort.Slice`.
4. **Dead code**: `profile/profile.go` imports `"sort"` but the only usage `var _ = sort.Strings` is a compiler-check import guard. The `isTech` function is an inline map lookup.
5. **Missing `context.Context` propagation**: Many engine methods don't accept context, preventing cancellation and deadlines.
6. **Config not actually consumed**: `config/config.go` defines a full Config struct and `Load` function, but the CLI never calls `config.Load()` — `openEngine()` creates the engine directly without loading TOML config.
7. **Duplicate entity extraction logic**: `engine/entities.go` (regex) and `engine/llm_entities.go` (LLM) are separate paths. The `LLMExtractor` falls back to regex on error, but the normal `Remember` path only uses regex.

### Missing Edge Cases

1. **Empty content handling**: `Remember` doesn't validate empty content.
2. **UUID collision**: No collision handling on node/edge UUIDs.
3. **Concurrent writes**: Relying on SQLite WAL + Go mutex, but `DualStream` and `Backbone` have separate locks.
4. **Very long content**: No truncation/budget check on content before storage.
5. **Special characters in FTS5**: No escaping of FTS5 special characters in `SearchNodes`.
6. **Cross-project ID conflicts**: No namespace prefix on node IDs.

## Summary

**Yaad is remarkably well-implemented for an early-stage project.** Despite the ambitious PLAN.md describing features spanning 6 phases, the vast majority of Phase 1-4 and Phase 6 features are implemented and tested. The core architecture is clean, the DAG implementation is solid, and the protocol support (MCP + REST + gRPC + SSE) is comprehensive.

**Key strengths:**
- 38 passing tests with good coverage of core paths
- Clean package structure with clear separation of concerns
- Advanced features like dual-stream ingestion, intent-aware retrieval, and semantic boundary detection
- Three SDKs (Go, Python, TypeScript)
- 27+ CLI commands, MCP server with tool profiles
- Privacy filtering, encryption, and version history

**Critical gaps:**
- `graph/rank.go` (PageRank) missing entirely
- `handleStale` is a placeholder — staleness detection not wired to servers
- Local embeddings are hash-based stubs, not semantic ONNX
- No `context.Context` propagation in engine
- No interfaces for storage/graph — tightly coupled
- All 38 tests in one file, no unit tests
